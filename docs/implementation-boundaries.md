# TrustIX 实现边界

当前代码已建立后端边界，并内嵌第一版 Web UI。

- `cmd/trustixd`：守护进程入口，负责装配控制面、路由、transport 和 dataplane；`-cleanup-dataplane-dry-run` 输出本机 TrustIX-managed TC/eBPF/VIP/sysctl/XDP cleanup JSON plan，`-cleanup-dataplane` 可在不启动 API 的情况下按 config/data-dir 执行清理，`-repair-dataplane` 会先清理残留再进入正常启动。
- `cmd/trustixctl`：运维 CLI 入口，覆盖状态、peer、member 查询/删除、trust show/policy/admins/roots/revoke/unrevoke、admissions list/show/pending/approve/approve-pending/revoke、route、route-policy、flow、endpoint、config 查询/按 seq 查看/验证/应用/回滚/snapshot/rejoin/restore-backup、capture 过滤、datapath、BPF maps、doctor 和多 Admin 证书签名管理请求命令边界；传入 Admin 证书时 GET 也会签名，供 host management API 的读认证使用。`-target-ix` 会把同一组管理命令通过本地 IX 的管理代理转发到目标 IX。
- `cmd/trustix-ca`：Root、Domain、Admin、IX 和 Route Authorization 证书工具入口；`verify` 会输出证书 SHA256 指纹，供 desired trust revocation 和 Admin allowlist 使用。
- `internal/core`：跨模块共享的 ID、prefix、resource path 等基础类型。
- `internal/config`：Desired Config schema 和基础校验，包括静态 peers/routes、bootstrap peer 配置、`lan.attach_mode: managed|existing`、`management.host_api` 主机侧管理入口、`management.tls` 管理 API/WebUI 共同 HTTPS 策略、`management.web_ui` 内嵌 WebUI 开关/custom override、`route_policy.import_prefixes/export_prefixes/dynamic_metric/import_transit_routes/transit_forwarding`、`transport_policy.mtu/fragment_policy`、`transport_policy.kernel_transport.mode`、公共 `transport_policy.crypto_placement`、本机 `kernel_modules.trustix_crypto`、`kernel_modules.trustix_datapath` 与 `kernel_modules.trustix_datapath_helpers` 生命周期声明，以及 `trust.revoked_cert_fingerprints`、`trust.trust_roots_pem`、`trust.admin_policy` 的规范化/校验。`existing` LAN attach 要求 gateway 已由系统配置在接口上，且不能开启 `manage_address`。公共 crypto placement 同时作用于 `tix_tcp` 和 `kernel_udp`；第一版 schema 只接受 `transport_policy.crypto_placement`，不接受 `transport_policy.tix_tcp.crypto_placement`。
- `internal/configlog`：签名配置事件、hash chain、Admin proof 审计字段和存储接口；正常同步只追加，管理员 rejoin 时可用已验签 snapshot 原子替换本地日志。`/domain/trust` 是 domain 级 trust resource，按 trust epoch / effective seq 生效，并携带完整 trust v2 policy；`/domain/admissions/<ix>` 是 domain 级 IX admission resource，按 Admin proof 批准或撤销 IX 加入。
- `internal/adminauth`：管理 API 写请求签名 envelope，使用 Admin 证书对 method、path/query、timestamp 和 body hash 签名；支持同一请求携带多组 Admin proof，同一 proof 可写入 config event 并由 peer 复验。
- `internal/pki`：TrustIX X.509 证书 profile、证书签发、metadata 解析和签名/验签。
- `internal/trust`：身份、证书、配置事件权限和路由授权验证接口。
- `internal/control`：peer 控制面消息、能力协商、配置同步、路由同步和状态 gossip 接口。
- `internal/routing`：prefix route table、LPM lookup、route owner/next-hop 分离、`unicast`/`local` route kind、route source/reason 元数据、flow stickiness 和 endpoint 选择接口。
- `internal/transport`：UDP、QUIC、TCP、WebSocket、HTTP CONNECT、TIX-TCP、GRE/IPIP tunnel 入口的插件式传输接口。
- `internal/transport/secure`：packet transport 加密 envelope，负责 IX 证书签名认证、crypto suite 协商、key source 选择、AEAD 加解密、epoch/sequence header 和 anti-replay。当前 userspace suite 支持 `AES-256-GCM-X25519`、`AES-128-GCM-X25519` 和 `CHACHA20-POLY1305-X25519`；默认只启用 AES-256-GCM，可用 `transport_policy.crypto_suites` 改成本机允许集合，并随 endpoint `security.crypto_suites` 广告传播。`transport_policy.encryption` 默认 `secure`，可设为 `plaintext`、`send_encrypted` 或 `receive_encrypted` 对 data packet envelope 做按方向开关；endpoint `security.encryption` 可覆盖全局策略并随成员广告传播，拨出端按对端声明自动选择互补方向。`plaintext + security.link_tls: required` 是 TLS-only data mode：TrustIX hello 和 IX 证书认证保留，data packet envelope 不再 AEAD，但 session 必须实际跑 LinkTLS，否则 fail-closed。key source 支持 `trustix_x25519` 和 `tls_exporter`：TLS exporter 只用于派生 TrustIX data key，不把 wire format 改成标准 TLS record；当底层 transport 实现 crypto offload session 且双向都加密时，握手后可把方向密钥、IV、epoch、key_source、suite 和 secure data wire-format 安装给 provider，由 provider 承担后续 packet AEAD。offload spec 的临时 key/IV clone 会在安装调用返回后清零；offload 成功后 secure session 会释放 userspace AEAD/IV 引用。当前 kernel crypto provider 接受 `AES-256-GCM-X25519` 和 `AES-128-GCM-X25519`；`CHACHA20-POLY1305-X25519` 会保留在 userspace，并在 kernel crypto map schema 中暴露 unsupported reason。
- `internal/daemon/transport_tls.go` / `transport_tls_status.go`：数据面底层 TLS 身份边界和运维诊断。默认用 IX 证书作为 transport TLS 证书；`transport_policy.tls_identity.mode: custom_cert` 可用独立 IP SAN/公网/私有 CA 证书做 TCP/WebSocket/HTTP CONNECT/QUIC 链路 TLS 和 TLS exporter。secure overlay 的 TrustIX hello 认证仍使用 IX 证书，不由自定义 transport 证书替代。`status`/`doctor` 会暴露 TLS key source、transport TLS identity、secure wire format、exporter-capable endpoint、`link_tls: required`/TLS-only endpoint 与 session 计数，以及 active session TLS 命中情况，并诊断强制 TLS exporter 但 endpoint 不支持 exporter、required/TLS-only session 未实际使用 LinkTLS、custom passive listener 缺证书等错配。
- `internal/daemon/nat.go`：IPv4 NAT/rewrite 第一版。`lan.mode: nat` 或 policy `rewrite: snat_gateway` 会把本地 LAN 出向 TCP/UDP/ICMP echo 源地址改写到 `lan.gateway`。出向 SNAT 已下推到 TC/eBPF fast path：daemon 通过 dataplane snapshot 同步 NAT source/route/exclude maps，TC 命中后改写 IPv4 源地址并更新 IPv4/TCP/UDP 校验和，capture event 带回原始源地址供 daemon 镜像限时 NAT state。NAT state 支持 `lan.nat.max_bindings`、`lan.nat.binding_ttl`、热更新重算过期时间、LRU 淘汰和状态计数；bindings 会增量同步到 TC/eBPF binding map。远端回包在解密后优先通过 NAT fast-path injector 交给 LAN egress TC 按 binding 做 DNAT，更新 IPv4/TCP/UDP 校验和并计入 `tc_nat_dnat_translations`；如果 fast path 不可用，daemon 仍按 NAT state DNAT 回原主机并重算 IPv4/TCP/UDP/ICMP 校验和。
- `internal/daemon/reject.go`：`routes[].kind: reject` 的回包语义。TC ingress 命中 reject route 时会计数并丢弃原包；对非 RST TCP 包，TC/eBPF 会直接构造固定 Ethernet/IPv4/TCP RST、刷新 IPv4/TCP checksum 并 redirect 回同一接口，计入 `tc_reject_tcp_rst_generated`，生成失败计入 `tc_reject_tcp_rst_errors` 后 fail-closed drop。对可接收错误的 ICMP 和非 TCP/ICMP IPv4 包，TC/eBPF 会构造固定 Ethernet/IPv4/ICMP destination unreachable、引用原 IPv4 头加 8 字节负载并 redirect 回同一接口，计入 `tc_reject_icmp_generated`，生成失败计入 `tc_reject_icmp_errors` 后 fail-closed drop。TCP RST 原包和不可接收 ICMP error reply 的 ICMP 类型在 TC 直接静默丢弃。IPv4 分片、IPv4 options 或短包仍 capture 到 daemon，由 LAN injector 回注本地 LAN。
- `internal/daemon/firewall_doctor_linux.go`：Linux firewall compatibility doctor。只读取 `/proc/net/*` 暴露 nftables/iptables 内核状态，不 shell out 到 `nft`/`iptables`，并声明 TrustIX 当前不管理防火墙规则。
- `internal/transport/tixtcp`：TCP-shaped TIX-TCP 数据面的共享帧格式、IPv4/TCP checksum 校验、crypto placement 选择和 secure crypto offload 接口，作为 userspace crypto 回灌 TC/XDP 和 eBPF kernel crypto 的稳定 payload contract。空配置的生产默认是 `userspace`；显式 `auto` 会优先使用 provider 报告 ready 的最快路径；请求 `kernel` placement 时必须由 dataplane provider 或 `.ko` AEAD device 明确支持并成功安装 flow crypto spec，否则拒绝启用；传给 dataplane installer 的临时 spec clone 会在安装调用返回后清零。
- `internal/transport/udp`：可收发 packet 的 UDP transport。默认普通 UDP socket 仍可用；当 dataplane 暴露 Kernel UDP provider 时，同一个 UDP endpoint 可切到 UDP/TIXU AF_XDP TX/RX fast path，并按公共 `transport_policy.crypto_placement` 选择 userspace、auto 或 kernel AEAD offload。空配置默认 `userspace`；显式 `auto` 优先选择内核后端，显式 `kernel` placement 在 provider 或 kernel crypto 不可用时 fail-closed，不静默回退。Kernel UDP session stats 默认把 `max_packet_size` 暴露为单个承载片可安全发送的大小，避免在有丢包链路上把 Linux TCP GSO super-packet 聚成大 TrustIX 分片包；需要实验时可用 `TRUSTIX_KERNEL_UDP_MAX_PACKET_SIZE` 显式放大逻辑包上限，最大仍限制在 IPv4 64KiB 以内。
- `internal/transport/quic`：可收发 packet 的 QUIC bidirectional stream transport。
- `internal/transport/tcp`：可收发 packet 的 TCP length-prefixed transport。
- `internal/transport/websocket`：可收发 packet 的 RFC 6455 binary frame transport。
- `internal/transport/httpconnect`：可收发 packet 的 HTTP CONNECT tunnel transport。
- `internal/transport/iptunnel`：GRE/IPIP kernel tunnel carrier transport。Linux 下通过 netlink 创建 GRE/IPIP tunnel netdev，并在隧道内使用固定 UDP carrier 封装 TrustIX packet；endpoint 字符串必须显式提供 `local=<underlay-ip>,remote=<underlay-ip>,local_carrier=<carrier-ip/prefix>,remote_carrier=<carrier-ip>[,port=<udp-port>,mtu=<bytes>]`。同一进程内按 `protocol+normalized config` 复用 tunnel netdev，并用 `<data-dir>/iptunnel/state.json` 记录 name/protocol/endpoint/role/config/ref_count；listen 与 dial 共享同一条 tunnel 时最后一个引用关闭才删除，crash 后由 `-cleanup-dataplane`/`-repair-dataplane` 读取该 state 删除残留 link。carrier 发送前会按配置 MTU fail-closed 拒绝 `payload+carrier header` 超限包，并在 session stats 暴露 `iptunnel_mtu`、`iptunnel_mtu_drops`、decode/drop 计数。它不使用 AF_INET raw socket，也不把任意 TrustIX envelope 字节伪装成 GRE/IPIP 内层 IP 包；非 Linux 或缺少 tunnel 能力时 fail-closed。
- `internal/runtime`：带 TTL 的 runtime state，不能作为 desired config。
- `internal/dataplane`：TC/eBPF map/program 管理、capture 订阅、packet policy 下推、packet 注入、local VIP 地址同步、kernel transport plane 状态接口、TIX-TCP 与 Kernel UDP fast-path/crypto placement 接口和 no-op 实现。
- `internal/dataplane/ebpf`：Linux dataplane 系统集成入口，使用 netlink 管理 link/address/TC qdisc 和 XDP attach，负责 LAN gateway、existing LAN iface 地址校验、管理 VIP `/32` 地址挂载/清理、sysctl、TC clsact qdisc、TC eBPF ingress/egress attach、stats map、IPv4 route LPM map、`transport_policy.mtu` / `fragment_policy: drop` 的 TC packet-policy fail-closed drop、`local` route 放行、route-hit perf capture、route-hit fail-closed drop、AF_PACKET LAN reinject、neighbor cache/event 同步、TIX-TCP 的 embedded C eBPF XDP redirect/AF_XDP TX-RX fast path、XDP 目的端口 allowlist map、XDP classifier counters、AF_XDP native/SKB attach 与 zero-copy/copy bind 协商、AF_XDP queue-count fallback、AF_XDP TX UMEM direct-build、TX flow/tuple queue affinity、TX bounded backpressure、AF_XDP RX UMEM direct parse、AF_XDP 收包 checksum fail-closed、kernel RX no-context deferral、flow TTL/GC、kernel crypto BTF/kfunc 与 `/proc/crypto` 能力探测、crypto fallback chain、能力状态和 snapshot state。`crypto_fallback.chain` 按 `.ko full_datapath`、`.ko skb/GSO helper`、TC/XDP/BPF direct、`.ko` AEAD device、userspace AEAD 排序，只有对应能力和驱动都 ready 才会成为 selected。TC route-hit capture 的 packet sample 窗口默认为 64KiB，用于覆盖常见 Linux TCP GSO super-packet；daemon 只转发完整 sample，超过窗口的包仍 fail-closed。对端解密/重组后回注 LAN 时，如果收到超过 LAN MTU 的普通 IPv4/TCP payload 包，会在 AF_PACKET 回注前按 MTU 做软件 TCP 分段、递增 TCP sequence、刷新 IPv4/TCP checksum；IPv4 options、已有分片、非 TCP、无 payload 或 SYN/RST/URG 等不安全形态继续返回 `MTU_EXCEEDED` fail-closed。`lan.attach_mode: managed` 默认启用 LAN offload protection：通过 SIOCETHTOOL ioctl 关闭 TC 前会破坏线性包假设的 GSO/GRO/TSO/partial-checksum 等 feature，并把原始状态写入 `<data-dir>/bpf/state.json`，退出/cleanup 时恢复；`existing` LAN 默认只告警不改口，除非设置 `TRUSTIX_LAN_OFFLOAD_PROTECTION=force|required`。`<data-dir>/bpf/state.json` 会记录 attach spec、是否添加 LAN address、是否准备 qdisc、local VIP、sysctl 原值、LAN offload 原值和 TIX-TCP XDP attach flags，供 crash 后 cleanup/repair 恢复。运行时不 shell out 到 `ip`/`tc`/`ethtool`。
- `internal/kernelmodule`：本机 Linux kernel module 生命周期边界。当前管理 `trustix_crypto`、`trustix_datapath` 和 `trustix_datapath_helpers` 三个第一版模块名，支持 `disabled`、`auto`、`required` 模式，使用 `/proc/modules`/`/sys/module` 探测状态，并通过 `finit_module`/`init_module`/`delete_module` syscall 加载/可选卸载，不 shell out 到 `insmod`/`rmmod`。`path: embedded` 会使用 release 二进制通过 Go embed 内置的 `.ko` payload；普通开发构建只带占位 payload，release 脚本会用 Go build overlay 替换成目标内核 `.ko`。加载 TrustIX `.ko` 时 daemon 会注入 `build_sha256=<target .ko sha256>`，状态读取 `loaded_sha256` 并按 `reload_on_upgrade` 判断是否需要安全重载；默认 `auto` 在已加载模块缺少指纹或指纹不同且 `ref_count=0` 时重载，busy 时报告 `upgrade_state` 和 reason。模块状态会读取 `abi_version` 与 `features`，归一成 `crypto_only`、`gso_skb`、`full_datapath` 等 capability tier；只有 `trustix_datapath` 可晋级 `full_datapath`，`trustix_datapath_helpers` 的基础 tier 仍是 `gso_skb`，route-TCP kfunc/XMIT 作为独立 feature flags 由选中策略和 runtime gate 再验证，未知模块名不会再根据 feature bit 推导能力，避免旧模块或第三方模块混入第一版能力链。它只执行本机 desired 配置，不会因为 peer gossip 自动在远端机器加载模块。
- `kernel/trustix_crypto`：可选 out-of-tree Linux module，用于在发行版内核已有 BPF crypto kfunc 但未注册 BPF `aead` type 时补齐 TrustIX AEAD-GCM 能力。默认模块通过 kernel crypto API 调用 `gcm(aes)`，底层 AES-NI/软件 provider 由内核选择；无硬件 AES 时会使用同步 generic `gcm(aes)` / `__gcm(aes)` 软件实现，也可用 `prefer_software=1` 在有 AES-NI 的机器上强制走软件路径做验证。模块 ABI v4 通过只读 sysfs 参数暴露 feature bitmask；当前实际实现的生产能力是 AEAD type、AEAD device、TC/XDP crypto-only direct kfunc、AES-NI/VAES direct crypto，不注册 skb/GSO/route/session helper kfunc，也不宣称 `gso_skb` 或 `full_datapath`。加载 `experimental_vaes=1` 后，x86 AES/AVX2/VAES/VPCLMULQDQ 主机会在 prepared mmap pool seal batch 上启用 TrustIX FPU-backed VAES/VPCLMUL 引擎；默认 `vaes_agg_ghash=1` 使用 4-block aggregated GHASH，`vaes_agg_ghash=0` 可回退旧 split GHASH loop，`vaes_fused_ghash=1` 保留正确但较慢的实验路径。模块初始化时如果探测到 VAES 不可用，会自动关闭 `experimental_vaes` / `experimental_vaes_kfunc`，避免每个 batch 反复进入不可用路径并增长 fallback 计数；`vaes_attempts`/`vaes_fallbacks` 用于确认实验路径是否实际命中。`kfunc_simd_fastpath` 第一版生产 hard-disabled：daemon 会剥掉该参数，模块 init 会强制 `N`，sysfs 只读。
- `kernel/trustix_datapath`：完整 kernel datapath 的独立 out-of-tree module 落点，拥有 `/dev/trustix_datapath`、datapath feature query ABI、route/session/session-wire/flow state ioctl、IPv4 hook、RX_STAGE/RX_WORKER 和 plaintext LAN TX 封装路径。加载 `enable_features=128 rx_worker_inject=1 tx_plaintext=1` 且自测通过时可报告安全 `full_datapath`；secure 加密和 GSO LAN TX 仍需要单独实现。第一版只允许 copied `netif_rx` RX worker 和 plaintext TX，stolen skb、direct/inline XMIT、RX stream/coalesce、checksum trust、MAC cache 等历史实验参数由 daemon 剥离，并由模块 init 强制关闭、sysfs 只读。
- `kernel/trustix_datapath_helpers`：skb/header/GSO 和 route-TCP helper kfunc 的独立 out-of-tree module，拥有 `/dev/trustix_datapath_helpers` 和 helper feature query ABI。当前第一版已经承载 TC skb/header/GSO helper kfunc、线性 route-TCP header helper，以及经生产 gate 验证的 route-GSO/XMIT helper 能力；默认可报告 `gso_skb` / `route_tcp_kfunc` / `route_tcp_xmit`，但 daemon 只有在 selected policy、runtime 参数、BTF/kfunc 能力和 cross-host gate 都匹配时才使用 route-GSO/XMIT 路径。即使 helper 误报 `full_datapath`，控制面也不会把它接纳为完整 datapath。outer-GSO batch、TIXT RX stream/coalesce 等仍未完成生产验证的路径继续 hard-disabled：daemon 按精确名和前缀剥离参数，模块 init 强制安全值，相关开关 sysfs 只读。
- `internal/daemon/membership.go`：动态 IX membership 和 route gossip，负责 signed advertisement 生成/验证、bootstrap peer 拉取/推送、成员表持久化、pending admission 持久化、TTL 过期清理、管理面成员删除、撤销、import/export prefix policy、`route_policy.import_transit_routes` 二跳动态路由导入开关、prefix 冲突仲裁、静态 transit route 的 owner 前缀占用、链上 admission 准入检查、动态 route/dataplane snapshot 热更新和 `/v1/route-policy` 决策输出。accepted 成员写入 `<data-dir>/members.json`，待批准广告写入 `<data-dir>/pending-members.json`；重启加载 pending 时会重新执行基础真实性、当前 trust/admission 和 TTL 校验。peer mTLS API 不接受成员删除。
- `internal/daemon/datapath.go`：userspace data path，订阅 TC route-hit packet、对进入 userspace 的 packet 保留 MTU/fragment policy 兜底校验、规范化 veth/TC 捕获到的 IPv4/TCP/UDP checksum、查路由、按 flow stickiness / least-conn policy 选择 endpoint、建立/复用 secure transport session、发送加密 packet；`transport_policy.kernel_transport.mode: require_kernel` 会过滤没有 kernel transport provider 的 endpoint。接收解密后的 packet 时，如果目的地址属于本地 LAN prefix 就注入 LAN，否则在 `route_policy.transit_forwarding` 允许时递减 IPv4 TTL 并按本机 route table 继续转发到下一跳 IX，从而支持显式 IX transit；关闭该开关只限制从其它 IX 收到后的二次转发，不影响本地 LAN route-hit 出站。热更新时只重建 data listeners/session，保留 capture forwarder，进程退出时会显式取消 capture subscription，避免 API/server 错误路径残留转发 goroutine；`datapath` API 暴露 `capture_forwarder_active`、`kernel_offload` placement 矩阵、`kernel_transport` 状态矩阵，并同时输出全局、per-route、per-peer 和 per-endpoint 计数。
- `internal/daemon/runtime_resources.go`：daemon 自身资源快照和 doctor 检查，输出 data-dir lock 持有状态/路径、Go heap/sys、goroutine、Linux RSS 和 open FD 计数，用于定位异常内存、文件描述符增长或重复 daemon 启动问题。
- `internal/daemon/state_files.go`：本机状态文件观测和 runtime cache 自愈边界。`status.state_files` 和 `doctor.state_files` 会检查 config log、`members.json`、`pending-members.json` 和 `iptunnel/state.json` 的路径、存在性、大小、mtime、权限、记录数、过期记录和 pending expiry 窗口，并暴露 config log backup 数量、保留上限和超限告警；`iptunnel/state.json` 会额外显示 active tunnel/ref 计数，便于确认 GRE/IPIP tunnel 生命周期是否跟随控制面清理。`config.log` 解码/链校验失败继续硬失败，`members.json` 和 `pending-members.json` 解码失败会隔离为 `.bad.<timestamp>` 后从空 runtime cache 启动，依赖后续 gossip/config sync 恢复。

- `scripts/linux-clean-release-smoke.sh`：干净测试机 release 全链路烟测。脚本会先清理 TrustIX 命名的 tmp、`tix-*` netns、`trustix_crypto.ko`、`trustix_datapath.ko` 与 `trustix_datapath_helpers.ko` 模块树残留，再从源码运行 `build-release-linux.sh`，用 `release-smoke-linux.sh` 验证 embedded `.ko` 自动加载/卸载、TCP TLS e2e 和 UDP/TCP/kernel_udp/tix_tcp NAT reverse e2e，用 release 包内预编译 `.ko` 跑 TrustIX kernel/full-datapath/helper module smoke，并默认继续跑 package 内的双 IX `kernel_udp`、NAT reverse `kernel_udp`、三 IX `kernel_udp` 与 `tix_tcp` kernel crypto e2e；每个阶段后都断言没有模块/netns/tmp 残留。如果目标 cloud 内核可以加载 `.ko` 和通过 device ioctl 使用 batch AEAD，但 BPF verifier 不支持 crypto kfunc provider selftest，kernel module smoke 会以能力不支持退出码返回，clean release 会继续验证 release 包和 kernel transport，并跳过严格 `tix_tcp` kernel crypto e2e。`TRUSTIX_CLEAN_RELEASE_SMOKE_NAT_REVERSE=0` 和 `TRUSTIX_CLEAN_RELEASE_SMOKE_3IX_KERNEL_UDP=0` 可关闭对应阶段。该脚本会删除 TrustIX 命名残留，只应在专用测试机执行。
- `internal/buildinfo` / `internal/buildassets`：构建元数据和嵌入资源摘要边界。`buildinfo` 只保存 ldflags 注入的版本信息和 Go/platform 信息；`buildassets` 汇总 eBPF `.o` 与 kernel module `.ko` 的 `embedded_kos` hash/size/ELF 状态，供 daemon status 和 `trustixd -version` 输出使用。

## TIX-TCP 目标边界

TIX-TCP 的配置、API、运行时协议身份、内核/BPF 统计键和测试环境变量统一使用 `tix_tcp`。旧协议名不再接受，跨此版本升级必须同步更新所有节点配置。TIX-TCP 不使用系统 TCP 传输数据包。目标模型是：

- 控制面可以用真实 TCP/TLS/mTLS 做认证、协商 flow id、epoch、cookie 和 key。
- 数据面是 TCP-shaped TIX-TCP frame，不进入 TCP socket buffer、TCP 重传队列或拥塞控制。
- 第一阶段使用 userspace crypto worker：TC/XDP 捕获或封装 frame，userspace 只做 AEAD seal/open，再通过 dataplane reinject 接口回到 TC/XDP 继续转发。
- 第二阶段保留 kernel crypto provider：同一 flow/frame contract 下，把 crypto placement 从 `userspace` 切到 `kernel`；secure transport 仍在 userspace 完成 IX 证书认证握手和 key source 派生，握手完成后把 send/recv AEAD key、IV、epoch、replay window、key_source 和 `trustix-secure-data-v1` wire format 安装给 provider，由 eBPF/kfunc 或其他内核 provider 完成后续 data packet AEAD 和 provider 侧 anti-replay。fast-path TX 会先解析 neighbor，再直接在 AF_XDP UMEM TX frame 中构建 Ethernet/IPv4/TCP/TIXT，按 `FlowID` 固定 TX queue，缺少 flow id 时按 underlay IPv4/TCP tuple 固定 queue，并在 TX frame pool 或 ring 短时压力下先 reclaim completion、kick TX、短等待重试；kernel placement 继续通过 XDP packet seal 程序（BPF_PROG_RUN）复用 provider ctx slot，并在 BPF 内完成 secure envelope、AEAD seal、IPv4/TCP checksum，把 seal 输出写回同一 UMEM frame，最后发布 TX descriptor。send ctx 会记录 last sequence 并拒绝重复/倒退 sequence，避免 nonce 复用。RX encrypted frame 已可在 attached XDP 中通过 map replacement 复用 provider ctx slot 完成 open/replay/drop，并重写 IPv4/TCP checksum 后再交给 AF_XDP；启动期 recv ctx 尚未安装时会先转交 AF_XDP/Go 侧短重试，而不是直接 no-context drop。
- 如果 kernel crypto 不可用，不能降级成未认证加密；应继续使用 userspace AES-GCM 或拒绝启用 `kernel` placement。
- Current `tix_tcp` is wired as the secure transport frame backend and Linux AF_XDP fast path. The attached XDP classifier is an embedded C eBPF object that parses fixed IPv4/TCP TrustIX frames, enforces the destination-port allowlist, updates XDP counters, and redirects authorized frames to AF_XDP. The loader negotiates native XDP + AF_XDP zero-copy first, then falls back to native copy and SKB copy; status exposes `xdp_attach_mode`, `af_xdp_bind_mode`, `zerocopy_enabled`, numeric provider counters, and fallback reason when a higher mode failed. The Go RX worker parses Ethernet/IPv4/TCP/TIXT directly from the AF_XDP UMEM frame view and copies only the final delivered payload before recycling the descriptor. Fast-path TX builds the TCP-shaped packet directly in the AF_XDP UMEM TX frame for both userspace and kernel crypto placement, selects TX queue by flow id or underlay tuple, and uses bounded completion reclaim/ring retry before reporting TX pool or ring pressure. When the optional kernel provider is ready, the loader switches RX to a kernel_crypto XDP object that map-replaces provider-owned flow/ctx maps; encrypted RX frames are opened in attached XDP, replay/header failures are dropped before AF_XDP, no-context startup races are deferred to AF_XDP/Go retry, and successful frames are marked kernel-opened with fresh IPv4/TCP checksums for userspace routing. TX also loads a BPF_PROG_RUN-compatible XDP packet seal object; the kernel placement fast path grows the direct-built frame by the secure-envelope overhead, seals payload with provider-owned send ctx, rewrites TIXT flags/length, and refreshes IPv4/TCP checksums directly into that same UMEM frame before the TX descriptor is published. `kernel_crypto_probe` reports kernel BTF, BPF crypto/RCU kfuncs, AES-GCM, AES-NI/CPU AES, AES-GCM software fallback availability, and runs an embedded bpfel ELF/CO-RE verifier selftest covering syscall ctx setup plus XDP/TC encrypt/decrypt kfunc paths. The probe also exposes provider map schema, selftest state, provider counters, flow-map lifecycle, a provider-side kptr ctx-map object, and synthetic AEAD-GCM `bpf_crypto_ctx_create` plus encrypt/decrypt roundtrip runtime probe when the verifier selftest passes. When the optional provider is present, `InstallTIXTCPCrypto` and `InstallKernelUDPCrypto` stage real flow keys into provider-owned kptr ctx slots and the shared flow map with separate protocol namespaces, then wipe Go-side temporary command/spec copies; production tix_tcp data frames can use AF_XDP UMEM direct-build/in-place TX XDP packet seal plus attached-XDP RX frame open with kernel AEAD, and UDP/TIXU can use the same provider for AES-GCM frame seal/open around the fixed UDP frame transport. Key material is not persisted, logged, or exported in API/status.

## kernel transport plane 目标边界

- `transport_policy.kernel_transport.mode` 是数据面 TX/RX 下沉策略，不是 TLS/QUIC/TCP 标准协议栈替代。`auto`/`prefer_kernel` 保持 socket transport 可用；`require_kernel` 会拒绝没有 kernel transport provider 的 endpoint。
- 当前可用 kernel transport provider 包括 `tix_tcp`、UDP/TIXU 以及 GRE/IPIP tunnel carrier。`tix_tcp` 提供固定 TrustIX TCP-shaped frame 的 XDP redirect、AF_XDP RX/TX、UMEM direct-build、flow/port map、health/status、碎片/重组 session 统计和可选 kernel crypto；UDP/TIXU 提供固定 IPv4/UDP TrustIX frame 的 XDP 目的端口 allowlist/redirect、AF_XDP RX/TX、flow/port map、health/status、AES-GCM kernel AEAD offload 和 userspace AEAD fallback。GRE/IPIP 第一版由 Linux kernel tunnel netdev 承担外层封装，TrustIX packet 在 tunnel 内使用 UDP carrier；secure 握手、证书、key 派生仍归 daemon，且没有 raw-socket fallback。
- 标准 QUIC 不做“内核化”承诺：QUIC TLS、ACK、loss recovery、congestion control、connection migration 和版本协商应继续由 userspace QUIC stack 承担。后续若需要 QUIC-like 低延迟数据面，应在 kernel transport plane 上定义固定 TrustIX frame，而不是把标准 QUIC 状态机塞进 eBPF。
- `transport_policy.kernel_transport.mode: require_kernel` 下，`tix_tcp` 和 UDP endpoint 只有在对应 provider 报告 available/reinject/fast_path 时才会进入候选；标准 TCP/QUIC/WebSocket/HTTP CONNECT endpoint 继续被视为 userspace socket transport。

## NAT/no-public endpoint boundary

- First version supports reverse reuse of an outbound data session. A NATed IX dials a reachable IX, completes the normal secure handshake with IX certificate identity, and the reachable side registers that inbound session as `reverse://inbound`.
- When sending back to that IX, datapath endpoint selection first reuses the registered reverse session. A peer endpoint may therefore be declared as `mode: passive` with no `address`; it becomes usable only after the NATed peer has established the outbound session.
- Dynamic route import still requires at least one enabled endpoint whose transport/security is usable locally. The endpoint no longer needs a dial address, but an unsupported transport is rejected as `no_usable_endpoint`.
- `trustixctl datapath` exposes reverse sessions with `address: "reverse://inbound"`, `reverse: true`, and `direction: "inbound_reverse"`. `trustixctl transports` marks no-address peer endpoints as `reverse_only` and reports `active_reverse_sessions` once an inbound session is available. `trustixctl doctor` includes `reverse_sessions=<n>` in the `data_path` detail.
- `TRUSTIX_E2E_NAT_REVERSE=1 sudo -E bash scripts/linux-e2e-smoke.sh` covers UDP by default. Setting `TRUSTIX_E2E_TRANSPORT=tcp|kernel_udp|tix_tcp` runs the same no-public reverse contract over TCP/TLS, UDP/TIXU AF_XDP, or tix_tcp AF_XDP; the kernel transports also assert the local listen port is present in the XDP allowed-port map even though no public address is advertised.
- This is not STUN, UDP hole punching, TURN, or automatic relay selection. At least one reachable/bootstrap IX must still be available for the NATed node to initiate control/data connectivity.
- Local passive endpoints only advertise an explicit `address`. `listen` remains a local bind address and is not treated as a public dial address.
